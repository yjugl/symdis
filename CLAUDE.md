## Project Status

All commands are fully implemented and the project is ready for real-life testing. The `disasm` command works end-to-end for Windows (PE), Linux (ELF), and macOS (Mach-O) modules: fetch sym+binary from symbol servers, find a function, disassemble, annotate with source lines/call targets/inlines/highlight, and print text or JSON output (`--format text|json`). Symbol coverage is NOT limited to Mozilla modules. On Windows: Microsoft system DLLs (ntdll, kernel32, kernelbase), kernel drivers (.sys), GPU drivers, and third-party vendor modules are supported â€” PDB files are fetched automatically from Microsoft, Intel, AMD, and NVIDIA symbol servers when `.sym` is unavailable. On Linux: system libraries are fetched via debuginfod, Snap Store, APT (Debian/Ubuntu), or pacman (Arch Linux). On macOS: binaries are extracted from .pkg archives. Tecken has `.sym` files for many non-Mozilla modules because Mozilla's crash infrastructure generates them from PDBs. Mach-O supports fat (universal) binaries with automatic arch selection. Binary fetch chain: Tecken â†’ Microsoft (Windows .dll/.exe/.sys) â†’ debuginfod (Linux, including openSUSE) â†’ Snap Store (Linux) â†’ APT (Linux, `--apt` + `--distro`; supports Ubuntu + Debian + derivatives via `--mirror`) â†’ pacman (Linux x86_64, `--pacman`; Arch Linux + derivatives via `--mirror`) â†’ FTP archive (Linux tar.xz, macOS PKG/XAR/cpio, Android APK). For Linux modules, the full ELF build ID for debuginfod/Snap/FTP lookups is extracted from the `INFO CODE_ID` record in the `.sym` file when `--code-id` is not provided; debuginfod/Snap/FTP require a real code_id (from `--code-id` or `INFO CODE_ID`) because the Breakpad debug ID only preserves 16 of the 20 build ID bytes and debuginfod requires an exact match. Windows kernel drivers (`.sys` files) are a first-class use case â€” PDB is fetched automatically, PUBLIC symbol names are resolved, and exact function bounds come from PE `.pdata`. Always provide `--code-file` for `.sys` files since the PDB-to-code-file heuristic defaults to `.dll`. When the sym file is unavailable, platform is inferred from code_id format (40-char hex â†’ Linux, 32-char hex â†’ macOS) so fallback sources are still attempted. The "Tried:" line in binary-not-found errors accurately lists only the sources that were actually attempted. Channel "default" (reported by some Linux distro builds in Socorro) is mapped to "release" for FTP archive lookup. Fenix (Firefox for Android) is supported via `--product fenix` with APK extraction. Focus (Firefox Focus for Android) is supported via `--product focus` with APK extraction. Ubuntu snap libraries are supported via the Snap Store backend (auto-detected or `--snap`). APT system libraries (libxml2, mesa, libdrm, libffi, etc.) are supported via the APT backend (`--apt` + `--distro`); supports Ubuntu (noble, jammy, ...), Debian (bookworm, bullseye, ...), and derivatives via `--mirror`/`--components`; downloads `.deb` packages, extracts binaries from `data.tar.{zst,xz,gz}`, and verifies build IDs; source package name is auto-detected from `.sym` source paths or specified explicitly via `--apt <package>`. Distro presets configure mirrors, components, and pocket suffixes automatically for known codenames. Arch Linux pacman packages are supported via the pacman backend (`--pacman`); downloads `.db.tar.gz` repo databases, matches packages by name or soname PROVIDES, downloads `.pkg.tar.zst` packages, extracts binaries, and verifies build IDs; use `--mirror` for Arch derivatives (CachyOS, Manjaro, EndeavourOS); x86_64 only (Arch ARM is a separate project). PDB symbol support: the `--pdb` flag skips `.sym` fetching and fetches PDB + binary instead (PDB replaces `.sym`, not the binary); without `--pdb`, PDB is auto-tried when `.sym` is unavailable for Windows modules. PDB fetch chain: cache â†’ Tecken (uncompressed + CAB) â†’ all remaining symbol servers in config (Microsoft, Intel, AMD, NVIDIA â€” all use symsrv protocol, uncompressed + CAB), using extended timeout (archive_timeout_seconds) for large PDBs. PDB data is converted into the existing `SymFile` struct so the annotation pipeline is unchanged. PDB inline frame tracking is supported via `InlineSiteSymbol` records and the IPI (ID information) stream; code ranges are extracted from binary annotations and inlinee names are resolved from the IPI stream. MSVC-decorated symbol names (`?Name@...`) are demangled via the `msvc-demangler` crate alongside Itanium ABI and Rust demangling. Source file paths from PDB are mapped to VCS paths (hg:..., git:...) via the srcsrv stream when available (Mozilla PDBs include srcsrv data for Mercurial and GitHub sources). Function names are enriched with MSVC-mangled names from public symbols where available (enables full demangled signatures with parameter types); inline call site file/line is resolved from the parent function's line records. PDB string table is loaded optionally â€” PDBs without one (common in newer Windows kernel modules) still yield functions/publics/inlines, just without source line annotations. PDB vs `.sym` trade-off: PDB files contain type information (class/struct layouts via TPI stream) that `.sym` files completely lack â€” this is PDB's key advantage, enabling the `field-layout` command. However, for disassembly, `.sym` is preferred: it has denser source line coverage (pdb crate panics on some modules in large PDBs â€” caught and skipped silently), and function name enrichment in PDB only covers ~37% of functions in large PDBs like xul.pdb. AI agents wanting type information (struct field offsets) should use PDB (`--pdb` or `field-layout`); agents that only need disassembly should use `.sym` (the default, lighter and faster). The `field-layout` command extracts C++ class/struct/union field layouts from PDB type information (TPI stream); shows fields sorted by offset with type names and sizes; `--offset` highlights the field at a given byte offset; `--fuzzy` enables substring matching on type names; forward-reference resolution ensures correct sizes for template instantiations like `RefPtr<T>` and `nsCOMPtr<T>`; works with Mozilla PDBs (xul.pdb, mozglue.pdb â€” 535K+ types) and some Microsoft public PDBs that include type info (e.g., ntdll.pdb has 686 types including `_RTL_CRITICAL_SECTION`, `_PEB`, `_TEB`); stripped/public-only PDBs produce a clear error about empty TPI. The `info` command reports PDB TPI availability: with `--pdb` it fetches and probes the PDB; without `--pdb` it checks the local cache only (no network cost). The `lookup`, `info`, and `fetch` commands are implemented. The `cache` command supports `path`, `size`, `clear` (with `--older-than`), and `list`. C++/Rust/MSVC symbol demangling is applied at display time (`--no-demangle` to disable). Configuration is loaded from a TOML config file, environment variables, and CLI flags with layered precedence (defaults < config file < env vars < CLI). Structured logging via `tracing` outputs to stderr at configurable verbosity (`-v` for info, `-vv` for debug). The `--offline` flag restricts operation to cached data only. HTML error page detection prevents caching corrupted downloads. Empty functions (size 0) are handled gracefully. PUBLIC symbol name lookup: `--function` searches PUBLIC symbols (with demangling) when no FUNC record matches, enabling disassembly of functions that only appear as PUBLIC symbols in PDB data (common for Windows kernel drivers). PE `.pdata` function bounds: when a binary is available, exact function sizes for PUBLIC symbols come from the PE `.pdata` (exception data) section instead of estimation from symbol distance. Indirect call resolution: `call/jmp [rip+disp]` (x86-64) and `call/jmp [absolute_va]` (x86 32-bit) are resolved to their IAT import names (e.g., `kernel32.dll!CreateFileW`); when the slot is not an import, on-disk pointer values are read and resolved as intra-module function pointer targets. ELF GOT import resolution: GOT slot VAs are mapped to import names from dynamic relocations (GLOB_DAT/JUMP_SLOT entries in `dynrelas`/`dynrels`/`pltrelocs`), enabling x86-64 `call [rip+disp]` through the GOT to resolve on Linux (e.g., `-fno-plt` builds). ELF and Mach-O `read_pointer_at_rva`: on-disk pointer values can be read for intra-module dispatch table resolution (vtables, function pointer arrays), complementing the PE-only implementation. PLT import resolution on ARM/AArch64 ELF: direct `bl <plt_addr>` calls are resolved to their import names (e.g., `memcpy`, `recvmsg`) instead of generic section names; architecture-specific PLT header sizes (32 bytes on AArch64, 20 bytes on ARM) ensure correct stub address computation. Mach-O stub import resolution: direct calls to `__stubs` entries on macOS (e.g., `call <__stubs+0x42>` on x86-64, `bl <__stubs+0x30>` on AArch64) are resolved to their import names (e.g., `libSystem.B.dylib!_malloc`) via the indirect symbol table; stub entry size and indirect symbol table index are read from the `S_SYMBOL_STUBS` section header. ARM32 Thumb-2 mode is auto-detected from ELF symbol metadata (mapping symbols `$t`/`$a` and function symbol Thumb bit); Capstone is configured in Thumb mode when the target function is in a Thumb region, producing correct disassembly for ARM32 binaries (including Fenix armeabi-v7a). AArch64 ADRP+LDR+BLR/BR indirect call resolution: the disassembly engine scans backward from `blr`/`br` instructions to find matching `adrp`+`ldr` sequences, computes the GOT/IAT slot address (`page + offset`), and sets `indirect_mem_addr` so the annotation pipeline resolves through `resolve_import` or `read_pointer_at_rva`; register clobber detection prevents incorrect resolution when the register chain is broken between instructions; this covers the standard indirect calling convention on AArch64 across all platforms (Linux ELF, macOS Mach-O, Windows PE, Android ELF). The `disasm` command supports `--socorro-json` to accept a full socorro crash report JSON (from file or stdin via `"-"`) in two modes: **frame mode** (default, no `--debug-file`) disassembles the function at the selected frame (default: frame 0, the crash frame) with offset and highlight auto-set from the frame; **module mode** (`--debug-file` + `--function`/`--offset`) disassembles a specific function in a different loaded module, with IDs resolved from `json_dump.modules` by `debug_file` name â€” this is frame-independent. In both modes, product, version, channel, build-id, distro, and backend flags (APT/pacman/snap) are auto-extracted; `--function`/`--offset` without `--debug-file` is rejected (the frame determines what to disassemble in frame mode). Linux distro detection from `lsb_release` auto-enables `--apt` (for known APT codenames) and `--pacman` (for Arch-like distros); Ubuntu snap runtime names are guessed from the codename (jammyâ†’gnome-42-2204-sdk+core22, nobleâ†’gnome-46-2404-sdk+core24) as a fallback when source-path detection doesn't find a snap name.

## Architecture

**Data flow for `disasm` command** (`commands/disasm.rs`):
1. Initialize tracing subscriber (verbosity from `-v`/`-vv`), resolve `Config` (TOML file + env vars + CLI flags), build HTTP client + cache
2. Concurrently fetch `.sym` file (or PDB with `--pdb`) and native binary (`tokio::join!`)
3. Parse `.sym` â†’ `SymFile` (or PDB â†’ `SymFile` via `symbols/pdb.rs`), parse binary â†’ `PeFile`, `ElfFile`, or `MachOFile` (via `BinaryFile` trait)
4. Find target function by name (FUNC â†’ PUBLIC â†’ exports) or offset (binary search); PUBLIC symbol sizes resolved from PE `.pdata` when available
5. Extract code bytes at function's RVA; detect ARM32 Thumb mode via `is_thumb()`; disassemble via Capstone
6. Annotate: source lines â†’ call targets (imports first, then sym; direct + RIP-relative indirect + PLT) â†’ inlines â†’ highlight (`disasm/annotate.rs`)
7. Demangle function name, call target names, and inline frame names (`demangle.rs`)
8. Format and print text or JSON output (dispatched via `--format`)

**Graceful degradation**: sym+binary â†’ full disassembly; binary-only â†’ raw disassembly; sym-only â†’ metadata without instructions; neither â†’ error.

**Key modules**:
- `config.rs` â€” TOML config file parsing (`toml 1.0`); `Config::resolve()` merges defaults < config file < env vars < CLI flags; `_NT_SYMBOL_PATH` parsing for cache dir; config file located via `SYMDIS_CONFIG` env var or platform default (`%APPDATA%\symdis\config.toml` / `~/.config/symdis/config.toml`)
- `cache.rs` â€” Flat WinDbg-compatible layout (`<root>/<file>/<id>/<file>`); atomic writes via tempfile; negative cache markers with configurable TTL; PDB cache support with separate miss markers (`-pdb.miss`)
- `fetch/` â€” Orchestrator: Tecken â†’ Microsoft (Windows PE) â†’ debuginfod (Linux) â†’ Snap Store (Linux) â†’ APT (Linux) â†’ pacman (Linux x86_64, Arch) â†’ FTP archive; PDB fetch chain: Tecken â†’ Microsoft/Intel/AMD/NVIDIA (all symsrv protocol); CAB-compressed downloads (`.dl_`/`.pd_` variants); server URLs parameterized from config; shared `compress_filename`/`decompress_cab` in `fetch/mod.rs`; `is_html_response()` detects corrupted downloads; offline mode skips network after cache miss; cache hit/miss logging via `tracing`; `fetch_pdb_file()` orchestrator for PDB fetch (Tecken â†’ loop remaining symbol servers using symsrv protocol) with extended timeout (archive_timeout_seconds) for large PDBs; `server_display_name()` maps server URLs to human-readable names for error messages
- `fetch/debuginfod.rs` â€” debuginfod client for Linux executables; server URLs from config (sourced from `DEBUGINFOD_URLS` env var or config file)
- `fetch/archive.rs` â€” Mozilla FTP archive client; downloads `.tar.xz` (Linux), `.pkg` (macOS), or `.apk` (Android/Fenix/Focus) build archives; extracts binaries; verifies ELF build IDs and Mach-O UUIDs; archive caching avoids re-downloading for multiple binaries from the same release
- `fetch/snap.rs` â€” Snap Store client for Ubuntu snap libraries; squashfs extraction via backhand; snap name auto-detection from sym file source paths
- `fetch/apt.rs` â€” APT package client for Ubuntu, Debian, and derivatives; `AptRepoConfig` type encapsulates mirror URLs, components, and pocket suffixes; `resolve_repo_config()` maps known codenames (noble, jammy, bookworm, bullseye, etc.) to preset configs; `custom_repo_config()` builds config from `--mirror`/`--components` flags; downloads Packages index (.xz with .gz fallback; cache key includes hostname for multi-mirror safety); searches configured pockets and components; resolves binary packages by Package: or Source: field; downloads `.deb` files (ar archives); extracts binaries from `data.tar.{zst,xz,gz}` via zstd/liblzma/flate2+tar; handles symlinks in tar (follows `libfoo.so.0` â†’ `libfoo.so.0.8000.0`); source package name auto-detected from `.sym` source paths (`/build/<name>-<random>/<name>-<version>/...` or `/usr/src/<name>-<version>/...` patterns, distinct from snap paths); Debian security pocket uses separate `security_base` URL

- `symbols/breakpad.rs` â€” Line-by-line parser for `.sym` files; functions and publics sorted by address for binary search; nameâ†’index HashMap for FUNC and PUBLIC exact lookup; `find_public_by_name`/`find_public_by_name_fuzzy` for PUBLIC symbol name search; `resolve_address` caps PUBLIC symbol distance at 64KB; `get_inline_at` returns active inline frames; `SymFileSummary` for lightweight scanning; `SymFile::from_parts()` constructor for PDB parser; `code_id` field parsed from `INFO CODE_ID` records (full ELF build ID for debuginfod/Snap/FTP lookups)
- `symbols/pdb.rs` â€” PDB â†’ `SymFile` converter via the `pdb` crate; extracts procedures, inline sites (via scope stack tracking), public symbols, and line programs per module; converts PDB section:offset addresses to RVAs via address map; inline code ranges extracted from `BinaryAnnotations` state machine; inlinee names resolved from IPI (ID information) stream; inline call site file/line resolved from parent function's line records; srcsrv stream parsed to map raw build paths to VCS paths (`hg:...`, `git:...`); function names enriched with MSVC-mangled names from public symbols (enables full demangled signatures); zero-size procedures become PUBLIC symbols; `catch_unwind` around each module to survive `pdb` crate panics (silent panic hook suppresses output)
- `symbols/tpi.rs` â€” PDB TPI stream parser for C++ type layout extraction; `extract_type_layout()` returns field offsets, sizes, type names, bitfield info, and base classes; `probe_type_info()` quickly checks whether a PDB has type information (used by `info` command); forward-reference resolution via nameâ†’TypeIndex map ensures correct sizes for template types; handles FieldList continuation chains, VirtualFunctionTablePointer, BaseClass/VirtualBaseClass; `catch_unwind` around TPI iteration for robustness
- `symbols/id_convert.rs` â€” Build ID â†’ Debug ID conversion (GUID byte-swapping for ELF); produces standard 33-char Breakpad debug IDs
- `binary/pe.rs` â€” Goblin-based PE parser implementing `BinaryFile` trait; RVA-to-file-offset via section table walk; IAT import resolution for call target annotation (uses `import.offset` for IAT slot RVA, NOT `import.rva` which is the Hint/Name Table RVA); `.pdata` exception entries parsed for exact function bounds (`function_bounds()` trait method); `read_pointer_at_rva()` reads on-disk pointer values for intra-module dispatch table resolution
- `binary/elf.rs` â€” Goblin-based ELF parser implementing `BinaryFile` trait; VA-to-offset via PT_LOAD segments; PLT import mapping for call target annotation with architecture-specific header sizes (AArch64: 32+16, ARM: 20+12, x86/x86_64: 16+16); searches `.plt` and `.plt.sec` sections (CET/IBT-enabled builds use `.plt.sec` as the actual call target); GOT import mapping from dynamic relocations (`dynrelas`/`dynrels`/`pltrelocs`) for GLOB_DAT/JUMP_SLOT entries â€” enables indirect call resolution through GOT slots (x86-64 `call [rip+disp]` in `-fno-plt` builds); `read_pointer_at_rva()` for intra-module dispatch table resolution; ARM32 Thumb/ARM mode detection via mapping symbols (`$t`/`$a`) and function symbol Thumb bit
- `binary/macho.rs` â€” Goblin-based Mach-O parser implementing `BinaryFile` trait; fat (universal) binary support with arch selection; VA-to-offset via segments; export/import resolution; `__stubs` import mapping via indirect symbol table for call target annotation (similar to ELF PLT); `read_pointer_at_rva()` for intra-module dispatch table resolution; UUID extraction for build verification
- `demangle.rs` â€” C++/Rust/MSVC symbol demangling via `cpp_demangle` (Itanium ABI, handles Mach-O `__Z` prefix natively), `rustc-demangle`, and `msvc-demangler` (MSVC ABI `?...` symbols); applied at display time, not at parse/storage time; import call targets in `dll!symbol` format are split before demangling; `--no-demangle` opt-out
- `disasm/engine.rs` â€” Capstone wrapper supporting x86/x86_64/ARM/ARM64 with Intel or ATT syntax; ARM32 Thumb mode via `thumb` parameter; extracts call targets from direct call/jmp instructions; computes memory slot RVAs for indirect call/jmp (x86-64 RIP-relative and x86 32-bit absolute addressing via x86 detail API); AArch64 ADRP+LDR+BLR/BR post-pass resolves indirect calls by scanning backward for matching instruction sequences and computing GOT/IAT slot addresses
- `disasm/annotate.rs` â€” Annotation pipeline: source lines, call target resolution (imports checked before sym for direct calls so PLT/Mach-O stubs resolve to import names; FUNC/PUBLIC/IAT/PLT/stubs + indirect via IAT/pointer tables for both x86-64 RIP-relative and x86 32-bit absolute addressing), inline frame tracking, highlight with mid-instruction range matching
- `output/text.rs` â€” Text formatter rendering source line comments, call target annotations, inline enter/exit markers, highlight (`==>`) marker; also defines shared `ModuleInfo`, `FunctionInfo`, `DataSource` types (including `BinaryAndPdb` and `PdbOnly` variants)

- `socorro.rs` â€” Socorro crash report JSON parser; serde structs for report/thread/frame/module/lsb_release; `resolve_crash_frame()` with `debug_file_override` parameter supports two modes: frame mode (resolve from crashing thread frame) and module mode (look up module directly via `find_module_in_list()` by `debug_file` name, frame-independent); auto-detects APT/pacman enablement from `lsb_release`; guesses snap runtime names for Ubuntu codenames (jammyâ†’gnome-42-2204-sdk+core22, nobleâ†’gnome-46-2404-sdk+core24)
- `commands/disasm.rs` â€” Primary command: concurrent sym+binary fetch, parse, find function, disassemble, annotate, format; `--socorro-json` two-mode support (frame mode vs module mode with `--debug-file` override); validates mode constraints (rejects `--function`/`--offset` without `--debug-file`); resolved `debug_file` from module takes precedence over CLI `--debug-file` (which serves as lookup key)
- `commands/lookup.rs` â€” Sym-file-only lookup by offset or function name; text and JSON output
- `commands/info.rs` â€” Module metadata using `SymFileSummary::scan`; PDB TPI probe for type info availability (`--pdb` fetches, otherwise cache-only); text and JSON output
- `commands/fetch.rs` â€” Cache pre-warming: concurrent sym+binary fetch with debuginfod and FTP fallback; `--pdb` skips `.sym` and fetches PDB + binary with TPI probe; text and JSON output
- `commands/field_layout.rs` â€” PDB type layout extraction: fetches PDB, extracts class/struct/union field layout from TPI stream via `symbols/tpi.rs`, offset matching, text and JSON output
- `commands/cache_cmd.rs` â€” Cache management: `path`, `size` (recursive walk), `clear` (with `--older-than` age filter), `list` (per-module artifacts)

## Key Conventions

- All commands are async (`async fn run`) and return `anyhow::Result<()>` with `.context()` for error messages
- `.sym` filename derivation: `xul.pdb` â†’ `xul.sym`, `libxul.so` â†’ `libxul.so.sym` (strip `.pdb` suffix, else append `.sym`)

- INLINE records in `.sym` files must follow their parent FUNC (before the next FUNC)
- Breakpad FUNC/PUBLIC may have optional `m` flag after keyword â€” parse and ignore
